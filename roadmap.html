Where we are

Done: per-position RegT calculator. 22 worked / derived examples from the Cboe Margin Manual (Nov 2021) are covered, plus guard, cash-account, and reconciliation tests. go test ./... passes with 30 test functions.

Each evaluation returns three numbers per the manual's distinction:

  • Requirement — gross "Margin Requirement"
  • AppliedProceeds — short-option premium credit
  • CashCall — net cash the customer must deposit

Current safeguards:

  • long_option_long_dated_listed is class-scoped to equity / equity ETF / index / volatility underlyings. ETN long-dated listed options correctly refuse until a specific rule exists.
  • is_limited_risk(legs) guard on generic_limited_risk_combo — net-short-call structures refuse to match instead of getting a fabricated finite MPL.
  • Cash-account coverage: short_put_uncovered, short_call_uncovered, vertical_spread, short_box_spread, short_strangle_or_straddle, generic_limited_risk_combo. Engine now returns numeric + DepositKind together.
  • Per-position reconciliation CLI exists: cmd/recon loads position/leg CSVs, runs the engine, and emits per-position diffs.

Roadmap

01 Engine correctness hardening next · highest priority

Before adding account/risk layers, close the remaining places where the engine can produce a confident but wrong number.

  • Fix mpl(legs) and is_limited_risk(legs): sample downside at U=0, treat only call upside as truly unbounded, and sum signed qty * mult instead of raw quantity.
  • Add tests for bounded put ratio spreads and mismatched contract multipliers.
  • Add position/rule input validation so missing qty, K, prices, underlying, or expiration cannot silently become zero-valued CEL inputs.
  • Rule-specific field validation. Convertible price, warrant price, and warrant K_equivalent are now enforced as type-intrinsic invariants in validateLeg. ETF price remains a hole: short_index_call_long_etf reads legs.le.price, so a leg with shares=10, price=0 still silently returns Requirement=0 there. Closing this systemically (rather than one Kind at a time) wants either per-rule required-field manifests in the YAML, or formulas that error on missing inputs instead of falling back to zero.
  • Add rulebook load validation: unique rule IDs, known legs_pattern, non-empty formula blocks, and deterministic slot signatures unless alternate constraint-aware binding is implemented.
02 Reconciliation v2: make bad data fail loudly next · high priority

The current recon harness is useful, but too forgiving for vendor reconciliation. Harden it before nightly runs or production data imports.

  • Validate required headers and duplicate headers up front.
  • Reject malformed, missing, NaN, and Inf numeric fields instead of defaulting them to zero.
  • Whitelist enums for account type, phase, side, kind, option type, style, venue, and settlement style.
  • Reject duplicate position_id rows and orphan legs whose position_id is not present in positions.csv.
  • Write permitted, deposit_kind, formula_key, and comparison basis. Let callers compare vendor values against gross Requirement or net CashCall.
  • Check CSV writer flush/close errors and format valid zeroes as 0.00, not blank cells.
03 Account aggregator (LMV / SMV / Equity / Adjusted Balance) next · after #1-#2

New Account type holding []Position, plus an Aggregate(account) function. Use Sterling's exact field names so reconciliation is apples-to-apples:

type AccountSnapshot struct {
    SODEquity        float64
    PnL              float64
    CurrentEquity    float64
    LMVStock         float64
    LMVOption        float64
    SMVStock         float64
    SMVOption        float64
    NetMV            float64   // LMV - |SMV|
    AdjustedBalance  float64   // Equity - LMV + |SMV Option|
    GrossExposure   float64
    StockLeverage   float64
    EquityRatio     float64   // Σ requirements / Equity
}

Unblocks every overlay and gives you the vocabulary to reconcile against vendor screens.

04 Account-level reconciliation next · after #3

The existing harness is per-position only. Once AccountSnapshot exists, extend recon to ingest vendor account fields and diff them alongside per-position rows. This is the point where nightly vendor comparison becomes meaningful.

05 Rule-set versioning + audit trail later

Result carries the rule-file SHA. A number from yesterday is reproducible. Cheap now, painful to retrofit. Persist evaluation inputs + outputs to a sqlite/parquet log.

06 House rule loading + policy overlays later

Split this into two separate capabilities: rulebook composition and post-rule add-ons.

  • Rulebook composition: load baseline Cboe rules plus firm-specific house_rules.yaml overrides/extensions with deterministic precedence.
  • Post-rule modifiers: vol add-on, low-price floor, market-cap add-on, and ADV / liquidity add-on.

Do not blur these. A house rule changes which rule/formula applies; an overlay modifies the result after the base regulatory number is known.

07 Risk-shock engine ("poor-man's PM") later

Delta-based scenario shocks — no vol surface needed:

  • Percent Stress: ±20% price, ±50% vol
  • Std-Dev Stress: ±3σ, ±20% vol
  • Single Name: ±50% price, ±50% vol
  • Worst Case = max of the three
  • + Liquidity Risk = 5% × position value × (qty / 30-day ADV) when qty > ADV

Gives a TIMS-shaped number well before there's a real TIMS. Useful as a sanity check.

08 Universal Spread Rule (cross-position decomp) later

Per Sterling: strategy matching has a uniqueness problem on complex portfolios. Replace it with optimal pair-up of shorts to longs as 2-leg spreads, sum the resulting requirements. This is also a prerequisite for proper portfolio-level RegT (multiple positions sharing offsets).

09 Full TIMS / pricing layer deferred · big swing

Real portfolio margin: OCC scenario grid, vol surface, binomial pricing for American options. Whole separate project. Defer until everything above reconciles against the vendor.

Things explicitly deferred

  • FLEX option exception handling — broker-policy-driven; not in CBOE manual's universal scope.
  • European long box cash treatment — relies on loan-value mechanism (margin-account only); no clean cash formula.
  • Day-trading / PDT — separate FINRA rule, separate engine surface.
  • Horizon / Pin / VaR views — Sterling has them but they're consumers of the same underlying state, not new engines.

Sequencing logic

The middle items (#5 – #8) are all valuable but only matter once a small set of foundations land:

  1. Engine correctness and input validation (#1).
  2. Reconciliation that fails on bad data instead of silently normalizing it (#2).
  3. Account vocabulary that matches the vendor (#3).
  4. Evidence that both position and account numbers agree with the vendor (#4).

With those four, every later piece is verifiable against an existing source of truth. Without them, building overlays, shocks, or decomp just produces unfalsifiable numbers.
